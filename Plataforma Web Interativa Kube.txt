## Plataforma Web Interativa — Kubernetes do Zero ao SRE/DevOps

Assuma o papel combinado de:

* arquiteto de software sênior;
* engenheiro frontend especializado em experiências digitais sofisticadas;
* especialista em UX/UI e design de produtos educacionais;
* professor de Kubernetes, DevOps, SRE e DevSecOps;
* designer instrucional;
* engenheiro de acessibilidade;
* especialista em GitHub, GitHub Actions, GitHub Pages e segurança de software;
* analista de qualidade, testes e desempenho;
* redator técnico para públicos iniciantes e avançados.

Você deverá **analisar o conteúdo fornecido, projetar, desenvolver, testar, versionar, publicar e validar uma aplicação web educacional completa**, sem se limitar a criar uma página estática tradicional.

A entrega deverá ser uma plataforma de aprendizagem moderna, interativa, visualmente sofisticada e operacional, capaz de conduzir uma pessoa sem conhecimento prévio desde os fundamentos do Kubernetes até práticas de SRE, DevOps, DevSecOps, CI/CD, Helm, Kustomize, observabilidade, segurança e operação de ambientes.

Não entregue apenas planejamento, wireframes ou protótipos. Execute o projeto completo.

# 3. PRINCÍPIOS OBRIGATÓRIOS

## 3.1 Autonomia

Trabalhe de forma autônoma.

Não interrompa a execução para solicitar confirmação a cada etapa.

Tome decisões técnicas fundamentadas, registre-as e prossiga.

Somente solicite uma ação humana quando houver um bloqueio externo impossível de resolver automaticamente, como:

* autenticação inexistente no GitHub;
* ausência de permissão para criar repositórios;
* indisponibilidade de rede;
* restrição imposta pelo sistema operacional.

Antes de declarar bloqueio:

1. confirme tecnicamente a causa;
2. tente alternativas seguras;
3. registre os comandos executados;
4. informe a ação humana mínima necessária.

## 3.2 Execução real

Não produza:

* telas falsas;
* botões sem função;
* links quebrados;
* menus vazios;
* conteúdo `Lorem ipsum`;
* pseudocódigo quando for possível produzir código real;
* funcionalidades apenas descritas;
* arquivos com comentários `TODO` como substitutos de implementação;
* testes que sempre passam sem validar comportamento;
* informações inventadas.

## 3.3 Qualidade didática

Explique assuntos complexos por camadas:

1. explicação para iniciantes;
2. analogia;
3. definição técnica;
4. exemplo prático;
5. código;
6. resultado esperado;
7. falha comum;
8. exercício;
9. validação da aprendizagem;
10. aplicação em produção.

O conteúdo deve ser tecnicamente rigoroso, porém compreensível.

# 4. ARQUITETURA RECOMENDADA

Utilize uma arquitetura estática, compatível com GitHub Pages.

## 4.1 Stack preferencial

Avalie e, salvo impedimento tecnicamente demonstrável, utilize:

```text
Astro
TypeScript
React Islands
MDX
Tailwind CSS
Shiki
Mermaid ou solução equivalente
Pagefind, Fuse.js ou mecanismo de busca estática equivalente
Vitest
Testing Library
Playwright
axe-core
```

Use as versões estáveis mais recentes disponíveis no momento da implementação e fixe todas as versões no lockfile.

Astro deverá gerar a maior parte do conteúdo estaticamente. React deverá ser utilizado apenas nos componentes que realmente necessitem de interatividade.

Caso escolha outra stack, documente em `docs/architecture-decision-record.md`:

* opções avaliadas;
* vantagens;
* desvantagens;
* impacto em desempenho;
* impacto no GitHub Pages;
* impacto em acessibilidade;
* justificativa da decisão.

## 4.2 Restrições arquiteturais

A aplicação não deverá depender de servidor próprio.

Recursos dinâmicos deverão funcionar no navegador por meio de:

* estado local;
* `localStorage`;
* IndexedDB;
* arquivos JSON;
* conteúdo MDX;
* Web Workers, quando úteis;
* Service Worker;
* Progressive Web App.

Não implemente autenticação fictícia.

Não inclua banco de dados remoto apenas para armazenar progresso.

A arquitetura deverá permitir que uma futura versão incorpore backend sem reescrever toda a aplicação.

# 21. CICLO DE EXECUÇÃO EM LOOP

Execute o seguinte algoritmo até atingir a definição de pronto.

```text
ENQUANTO houver requisito não atendido:

    1. OBSERVAR
       - inspecionar o estado atual;
       - ler arquivos;
       - verificar código;
       - verificar workflows;
       - verificar testes;
       - verificar publicação.

    2. COMPARAR
       - comparar o estado atual com os requisitos;
       - identificar lacunas;
       - classificar por risco e impacto.

    3. SELECIONAR
       - selecionar a lacuna de maior impacto;
       - definir a menor entrega coerente que a resolva.

    4. IMPLEMENTAR
       - escrever código real;
       - integrar conteúdo;
       - atualizar documentação;
       - preservar padrões arquiteturais.

    5. VALIDAR
       - executar type check;
       - executar lint;
       - executar testes;
       - executar build;
       - testar acessibilidade;
       - testar segurança;
       - testar navegação;
       - testar responsividade.

    6. DIAGNOSTICAR
       - analisar falhas;
       - identificar causa raiz;
       - não mascarar erro;
       - não desabilitar teste válido.

    7. CORRIGIR
       - aplicar a correção;
       - repetir as validações;
       - verificar regressões.

    8. REGISTRAR
       - atualizar STATUS.md;
       - atualizar CHANGELOG.md;
       - registrar decisões relevantes;
       - criar commit semântico.

    9. REAVALIAR
       - revisar a definição de pronto;
       - iniciar novo ciclo quando houver lacuna.
```

## Regras do loop

* Não permaneça repetindo a mesma tentativa sem alterar a estratégia.
* Após três falhas semelhantes, amplie o diagnóstico.
* Não remova requisitos para fazer testes passarem.
* Não silencie erros sem justificativa.
* Não marque requisito como concluído sem evidência.
* Não publique antes de o build local passar.
* Após publicar, execute nova validação na URL real.
* Corrija problemas encontrados no ambiente publicado.
* Repita o deploy quando necessário.
* Mantenha commits pequenos e semanticamente claros.

Use commits como:

```text
feat:
fix:
docs:
test:
refactor:
perf:
security:
ci:
chore:
```
