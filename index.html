1. Fundamentos para Leigos

O que é um Container
Explicação Simples

Um container é uma forma de empacotar uma aplicação com suas dependências para que ela rode de maneira previsível. Ele não é uma máquina virtual completa; normalmente é um processo isolado que compartilha o kernel do sistema operacional.

Explicação Técnica

Um container usa namespaces e cgroups do Linux para isolar processos, rede e recursos. Ele compartilha o kernel do host mas tem seu próprio filesystem, rede e espaço de processos.

🛠️ Analogia Prática

Pense em uma marmita lacrada. Ela já contém tudo que você precisa para aquela refeição. A imagem do container é a receita pronta; o container em execução é a marmita sendo consumida.

O que é uma Imagem
Explicação Simples

Imagem é o pacote imutável a partir do qual containers são criados. Kubernetes não compila sua aplicação — ele baixa uma imagem de um registry e executa essa imagem em um Pod.

Exemplo
YAML
containers:
  - name: backend
    image: ghcr.io/minha-org/backend:v1.0.0
Boa prática: Evite usar a tag latest em produção. Prefira tags versionadas ou digest de imagem para garantir previsibilidade e rollback confiável.
O que é um Registry
Explicação Simples

Registry é um repositório de imagens. Ele armazena e distribui as imagens usadas no Kubernetes.

O que é Kubernetes
Explicação Simples

Kubernetes é uma plataforma de orquestração. Ele recebe declarações do estado desejado e tenta manter o ambiente de acordo com esse estado. Em vez de entrar no servidor e iniciar processos manualmente, você declara recursos como Deployment, Service e Ingress.

Explicação Técnica

Kubernetes implementa um ciclo de reconciliação contínuo: você declara o estado desejado (via YAML/manifestos), e o control plane compara com o estado real, executando ações para convergir o estado atual ao desejado.

O que é K3s
Explicação Simples

K3s é uma distribuição Kubernetes leve. Ele é muito usado em laboratórios, edge, VMs pequenas, homelabs e cenários em que você quer simplicidade operacional. Ele reduz complexidade e vem com padrões prontos, mas continua sendo Kubernetes.

Por que Kubernetes existe

Antes do Kubernetes, para colocar uma aplicação no ar, um administrador precisava acessar o servidor, instalar dependências, configurar serviços, iniciar processos e monitorar manualmente. Se o servidor caísse, alguém precisava restaurar tudo.

Kubernetes muda completamente esse paradigma: você declara o que quer (via arquivos YAML) e o Kubernetes cuida de executar, manter, escalar e recuperar automaticamente.

Conceito-chave: Kubernetes usa um modelo declarativo. Você diz "quero 3 réplicas do backend rodando com essa imagem", e o cluster mantém essa condição. Se um Pod cair, outro é criado automaticamente.

3. Recursos Básicos

Pod

A menor unidade implantável. Normalmente contém um container principal, mas pode conter sidecars (agente de log, proxy local).

Unidade básica

Deployment

Usado para aplicações sem estado. Cria ReplicaSets, que criam Pods. Realiza rollout gradual em atualizações.

Controlador principal

ReplicaSet

Garante que exista um número desejado de Pods. Você raramente cria diretamente — o Deployment cria e gerencia para você.

Gerenciado

StatefulSet

Para aplicações com identidade e estado persistente (bancos de dados). Cria Pods com nomes previsíveis (postgres-0, postgres-1).

Estado persistente

DaemonSet

Garante que um Pod rode em todos os nodes. Usado para agentes de monitoramento, logs e segurança.

Um por node

Job / CronJob

Job executa uma tarefa até terminar. CronJob executa tarefas em agenda (ex: backup todo dia às 02h).

Tarefas

Exemplo de Deployment

YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: applications
spec:
  replicas: 2
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
        - name: web
          image: nginx:alpine
          ports:
            - containerPort: 80

5. Services e Portas

Por que Service existe
Explicação Simples

Service é como o número fixo de telefone de uma aplicação. Os Pods podem mudar de IP, mas o Service continua apontando para eles.

Explicação Técnica

Service cria um ponto estável de rede para um conjunto de Pods selecionados por labels. O kube-proxy cria regras de encaminhamento (iptables/IPVS) para distribuir tráfego entre os Pods.

Portas Explicadas

CampoOnde existeExplicação
containerPortContainer/PodPorta em que a aplicação escuta dentro do container
targetPortServicePorta do Pod para onde o Service envia tráfego
portServicePorta do Service acessada pelos clientes internos
nodePortNodePorta aberta no node para acesso externo
hostPortPod/HostPorta do host mapeada diretamente para o Pod

Tipos de Service

ClusterIP

Uso interno. Ideal para backend, banco e comunicação entre aplicações. Acesso apenas dentro do cluster.

Interno

NodePort

Abre porta alta no node. Bom para laboratório, mas não é ideal para exposição pública elegante.

Porta no node

LoadBalancer

Pede um balanceador externo. Ideal em cloud ou com MetalLB em bare metal.

Externo

HostPort

Mapeia porta do host para o Pod. Útil em ambientes pequenos, mas exige cuidado.

Cuidado

Ingress

Roteia HTTP/HTTPS por domínio e caminho. Ideal para aplicações web.

HTTP/HTTPS

ExternalName / Headless

ExternalName aponta para DNS externo. Headless retorna IPs dos Pods diretamente.

Especial

Exemplos

ClusterIP ▼
YAML
apiVersion: v1
kind: Service
metadata:
  name: backend
  namespace: applications
spec:
  type: ClusterIP
  selector:
    app: backend

kube-proxy e CNI

kube-proxy
Explicação Simples

kube-proxy cria regras de rede para que Services encaminhem tráfego aos Pods corretos.

Explicação Técnica

Em muitos clusters, ele usa iptables ou IPVS para implementar esse encaminhamento. Cada Service ClusterIP vira uma regra que distribui pacotes entre os Pods endpoints.

Cliente → Service ClusterIP → kube-proxy → Pod A ou Pod B

CNI — Container Network Interface

CNI é a camada que permite que Pods tenham IP e conversem entre si.

CNICaracterísticaQuando considerar
FlannelSimples e leveK3s, laboratório, ambientes pequenos
CalicoNetworkPolicy forteAmbientes com controle de rede
CiliumeBPF e observabilidadeAmbientes avançados e alta performance
AntreaOpen vSwitchAmbientes VMware/on-premises

6. Ingress Controllers

O que é Ingress
Explicação Simples

Ingress é a regra que diz: quando chegar uma requisição HTTP/HTTPS para determinado host/path, encaminhe para determinado Service.

Explicação Técnica

Ingress é apenas uma declaração. Quem realmente escuta portas e aplica regras é o Ingress Controller (Traefik, NGINX, HAProxy ou Envoy).

YAML
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app
  namespace: applications
spec:
  ingressClassName: traefik
  rules:
    - host: app.seu-dominio.com.br
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend
                port:
                  number: 80

7. Storage e Volumes

Por que volume é necessário
Explicação Simples

Containers são efêmeros. Se você salvar dados apenas dentro do container, pode perdê-los quando o Pod for recriado. Por isso bancos e arquivos precisam de volumes persistentes.

Fluxo de Storage
Pod → PVC → StorageClass → PV → Disco

Tipos de Volume

TipoPersistente?Uso Recomendado
emptyDirNãoCache temporário dentro do ciclo de vida do Pod
configMapNão é storageMontar arquivos de configuração
secretNão é storageMontar senhas/certificados/tokens
hostPathDepende do nodeLaboratório; cuidado em produção
local-pathSim (local)K3s, laboratório simples
NFSSimCompartilhamento entre Pods
LonghornSim (distribuído)Homelab, edge, clusters com múltiplos nodes
Ceph/RookSim (distribuído)Produção on-premises robusta

Exemplo de PVC

YAML
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-data
  namespace: applications
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi

Access Modes

ModoSignificadoUso Comum
ReadWriteOnceUm node monta para escritaBanco comum
ReadOnlyManyMuitos leemConteúdo estático compartilhado
ReadWriteManyMuitos leem e escrevemNFS, compartilhamento
ReadWriteOncePodApenas um Pod escreveControle restrito de banco

Longhorn e Soluções de Storage

Longhorn
Explicação Simples

Longhorn é uma solução de storage distribuído para Kubernetes. Ele replica volumes entre nodes e permite snapshots, backups e recuperação.

Quando usar

local-path

Laboratório e ambientes simples. Grava no disco local do node.

Simples

Longhorn

Homelab, edge, pequenos clusters com múltiplos nodes e necessidade de persistência.

Distribuído

NFS

Compartilhamento simples entre múltiplos Pods. Comum com NAS.

Compartilhado

Ceph/Rook

Ambiente on-premises robusto e produção madura.

Enterprise
Use Longhorn quandoTenha cuidado quando
Precisa de replicação de volumesNodes têm pouca memória
Quer snapshots e backupsRede entre nodes é instável
Tem mais de um nodeDiscos são muito lentos
Quer painel visual de storageAmbiente é mínimo com recursos apertados

          ports:
            - name: http
              containerPort: 8080
          envFrom:
            - configMapRef:
                name: backend-config
            - secretRef:
                name: backend-secret
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 512Mi
          readinessProbe:
            httpGet:
              path: /health
              port: http
          livenessProbe:
            httpGet:
              path: /health
              port: http
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop:
                - ALL

Por que cada campo existe

CampoMotivo
replicasDefine disponibilidade e capacidade básica
selector/matchLabelsLiga Deployment aos Pods corretos
imageDefine o artefato que será executado
portsDocumenta portas do container
envFromCarrega configurações e segredos
resourcesEvita consumir o node inteiro e ajuda o scheduler
readinessProbeSó recebe tráfego quando pronto
livenessProbeReinicia se travar
securityContextReduz privilégios do container
Boa prática: Sempre defina resources.requests e resources.limits. Sem isso, o scheduler pode agendar Pods que consumam todos os recursos do node.

11. Segurança Operacional

Segredos

Nunca coloque senha em ConfigMap, Dockerfile ou código. Use Secret, preferencialmente integrado a um cofre externo em ambientes maiores.

Boas Práticas de Pod Security

  • Executar como usuário não-root
  • Desabilitar escalada de privilégios
  • Remover capabilities desnecessárias
  • Usar filesystem somente leitura quando possível
  • Definir requests e limits
  • Evitar imagens desconhecidas e tags latest

15. Troubleshooting — Metodologia SRE

Cadeia de Diagnóstico

Fluxo de diagnóstico (do mais externo ao mais interno)
DNS → CDN/CF → Firewall → :80/:443 → Traefik → Ingress → Service → Endpoints → Pod → Logs

Erros Comuns

CrashLoopBackOff

Causa provável: aplicação inicia e cai repetidamente.

Diagnóstico
kubectl logs POD -n applications
kubectl logs POD -n applications --previous
kubectl describe pod POD -n applications

ImagePullBackOff

Causa provável: imagem errada ou registry privado sem credenciais.

Diagnóstico
kubectl describe pod POD -n applications
# Verificar: nome da imagem, tag, imagePullSecrets

Pod Pending

Causa provável: recursos insuficientes ou volume pendente.

CausaComo identificarComo corrigir
Falta de recursoEventos: Insufficient cpu/memoryReduzir replicas/requests ou aumentar node
Volume pendentePVC PendingVerificar StorageClass
nodeSelector errado0/2 nodes availableAjustar labels ou nodeSelector
Taint sem tolerationEventos citam taintAdicionar toleration ou remover taint

Cloudflare 521 — Origin Inacessível

Cloudflare não conseguiu conectar ao origin. Verifique se 80/443 estão abertos.

Diagnóstico
sudo ss -tulpn | grep -E ':(80|443)\b'
curl -vk https://IP_PUBLICO -H "Host: app.seu-dominio.com.br"

Cloudflare 526 — Certificado Inválido

Certificado inválido no origin em modo Full Strict.

Diagnóstico
kubectl get secret -n traefik
kubectl get tlsstore -n traefik
openssl x509 -in cloudflare.crt -text -noout

Erro 502 — Bad Gateway

Service existe, mas endpoints estão vazios ou Pod não responde na porta esperada.

20. Glossário

kube-apiserver

API central do cluster — a "portaria" que recebe todas as requisições.
Endpoint RESTful que valida e processa objetos Kubernetes.

etcd

Banco de dados chave-valor que guarda todo o estado do cluster.
Store distribuído, consistente e tolerante a falhas (RAFT).

kube-scheduler

Escolhe em qual node cada Pod vai rodar.
Algoritmo de agendamento considera recursos, taints, afinidade e affinidade.

kube-controller-manager

Supervisor que verifica se o estado real bate com o desejado.
Executa loops de reconciliação para Deployment, ReplicaSet, Node, etc.

kubelet

Agente que roda em cada node e cuida dos Pods.
Garante que os containers descritos nos Pods estejam rodando e saudáveis.

kube-proxy

Cria regras de rede para encaminhar tráfego dos Services aos Pods.
Usa iptables ou IPVS para implementar Service networking.

CNI

A "placa de rede" dos Pods — dá IP e conectividade.
Container Network Interface: plugin que configura a rede dos Pods (Flannel, Calico, Cilium).

CRI

Interface que permite ao Kubernetes usar diferentes runtimes de container (containerd, CRI-O).
Container Runtime Interface: padrao que substituiu a integracao direta com Docker.

CSI

Interface para conectar Kubernetes a soluções de armazenamento.
Container Storage Interface: permite drivers de storage como Longhorn, NFS, etc.

Pod

A menor unidade que Kubernetes consegue gerenciar.
Agrupa 1+ containers com rede e volumes compartilhados.

Deployment

Controlador que mantém N réplicas da sua aplicação rodando.
Cria ReplicaSets e gerencia rollout/rollback de versões.

Service

Endereço estável e fixo para acessar um conjunto de Pods.
Agrupa endpoints por labels e distribui tráfego (ClusterIP, NodePort, LB).

Ingress

Regra que diz: "este domínio/caminho vai para este Service".
Objeto networking.k8s.io que define rotas HTTP/HTTPS baseadas em host e path.

Ingress Controller

O "motor" que escuta portas e aplica as regras do Ingress.
Software como Traefik, NGINX, HAProxy que implementa o protocolo Ingress.

Namespace

Uma "pasta" lógica para organizar recursos no cluster.
Isolamento virtual de recursos, útil para multi-tenant ou separar ambientes.

ConfigMap

Armazena configurações não sensíveis (chave=valor ou arquivos).
Objeto que injeta variáveis de ambiente ou monta arquivos no Pod.

Secret

Armazena dados sensíveis (senhas, tokens, certificados).
Base64-encoded; para melhor segurança use Sealed Secrets ou Vault.

PVC (PersistentVolumeClaim)

Um "pedido" de armazenamento que a aplicação faz ao cluster.
Solicita volume com determinada capacidade e access mode.

PV (PersistentVolume)

O volume de armazenamento real provisionado no cluster.
Recurso cluster-wide que representa um pedaço de storage.

StorageClass

Define "como" provisionar volumes (local, NFS, Longhorn, etc.).
Classe com provisionador CSI, parâmetros e política de reclamation.

Helm

Gerenciador de pacotes do Kubernetes — o "apt-get" do K8s.
Templates parametrizados (Charts) que empacotam manifests complexos.

Chart

O pacote do Helm: um conjunto de templates + valores.
Diretório com templates YAML, Chart.yaml e values.yaml.

Release

Uma instalação de um Chart com configurações específicas.
Instância nomeada de um Chart aplicado ao cluster.

GitOps

Usar o Git como "fonte da verdade" para toda a infraestrutura.
Fluxo onde o cluster observa o Git e aplica automaticamente mudanças.

IaC

Infraestrutura como Código: criar servidores com arquivos de configuração.
Ferramentas como Terraform e Ansible que tornam infra reproduzível.

HPA

Autoscaler horizontal: aumenta/reduz réplicas automaticamente.
HorizontalPodAutoscaler: escala com base em CPU, memória ou métricas custom.

PDB

Garante que nem todos os Pods fiquem fora ao mesmo tempo.
PodDisruptionBudget: limita interrupções voluntárias (drain, upgrade).

RBAC

Controle de quem pode fazer o quê no cluster.
Role-Based Access Control: Roles, ClusterRoles, Bindings.

NetworkPolicy

Firewall entre Pods: define quem pode falar com quem.
Políticas de ingress/egress baseadas em labels e ports.

Longhorn

Storage distribuído com replicação, snapshots e backups.
CSI driver CNCF para Kubernetes com UI web e HA.

MetalLB

Balanceador de carga para Kubernetes bare metal.
Permite Service type LoadBalancer sem cloud provider.

Traefik

Proxy reverso e Ingress Controller moderno e automático.
Descobre Services/Pods automaticamente via Kubernetes API.

NGINX Ingress

O Ingress Controller mais maduro e amplamente usado.
Implementação do Ingress baseada no NGINX.

Cloudflare

CDN/WAF/DNS que protege e acelera sua aplicação na borda.
Proxy reverso com TLS, cache, DDoS protection e rate limiting.

LoadBalancer

Tipo de Service que pede um IP externo ao provedor.
Service que expõe tráfego via balanceador externo (cloud ou MetalLB).

NodePort

Tipo de Service que abre uma porta fixa em cada node.
Range 30000-32767, acessível pelo IP de qualquer node.

ClusterIP

Tipo de Service interno — só acessível dentro do cluster.
IP virtual que distribui tráfego entre os Pods endpoints.

HostPort

Mapeia uma porta do node diretamente para o Pod.
Porta do host exposta externamente sem Service.
